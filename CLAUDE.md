## 二、铁律（不可违背）

1. **不准编造**：没实跑就写"未实测"，不写"已验证"。所有核心数值必须来自真实代码/数据/推导，可复算。可分解结果写"近似最优解"，不写"全局最优"。
2. **算法服帖题目，不加模戏**：每用一个算法说清——为什么适合本题、输入输出、规模限制、如何验证。禁止为显高级乱堆神经网络/遗传/退火/粒子群/TOPSIS/灰色预测。
3. **闭环必须完整**：问题分析 → 假设 → 符号 → 模型 → 算法 → 结果 → 检验 → 优缺点。不只给公式不解释变量；不只给结果不解释合理性；不只写算法名不给求解流程。
4. **语言像投稿论文，不像 AI 作文**：少用"具有重要意义""效果显著""充分体现"。多用"本文首先…然后…最后…""由表可知…""结果表明…""这说明…"。
5. **量子比特消耗最小化（题目硬性要求）**：题目原文「采用恰当的变量定义方式和惩罚方法，**在尽可能减少量子比特消耗的前提下**，将其转化为 QUBO 形式」——这是 Q1/Q2/Q3/Q4 都明列的赤字约束。每次设计 QUBO 编码、做实验、写论文段落，都必须把"量子比特用量"作为**首要正交目标**与"目标函数最优性"并列处理：
   - **设计阶段**：每提出一种编码必须明确写出 `nvar = f(n, ...)` 公式，并与等价编码（如 binary 编码 vs one-hot vs 时间槽线性化）比较比特数；非必要不引入辅助比特；惩罚项尽可能从约束消去而非引入 slack 变量。
   - **实验阶段**：每个 JSON / 实验日志条目必须显式记录 `n_qubo_vars`（QUBO 变量数）+ `n_ising_spins`（含辅助自旋数），并在论文图/表中至少出现一次"比特数 vs 规模 n"的曲线或表格。
   - **论文写作**：摘要、模型章节、结果章节都要回应这一约束——典型句式："本文采用 one-hot 位置编码，n=15 客户共 n²=225 比特；若改用 binary 编码可降至 n·⌈log₂n⌉=60 比特，但会破坏 QUBO 二次性，故权衡后保留 one-hot。"
   - **比较时优先比"比特数 / cost / 可行率"三联表**，而非只比 cost 一项。Q3、Q4 的大规模分解策略**必须**给出"分解后单子问题比特数 ≤ 真机 550 比特上限"的论证。
6. **SDK 验证算法必须为最终真机服务（链路一致性）**：题目要求 Q3、Q4 的最终求解上"玻色 550W 相干伊辛真机"。Kaiwu SDK 内置经典 SA 虽然没有 550 比特限制（可硬喂 2500 维 QUBO），但**绝对禁止**用 SDK 跑一种"通不过真机"的算法只为得到漂亮数值——SDK 阶段必须采用与真机**完全同套的编码 + 比特预算 + 8-bit 精度约束**，仅算力后端从 SA 切到 CIM。具体要求：
   - **比特预算硬上限**：SDK 阶段每次喂给 `kw.classical.SimulatedAnnealingOptimizer.solve(ising_matrix)` 的 ising 维度必须 ≤ 550（含辅助自旋）。即对应 QUBO 维度 ≤ 549。**禁止**让 SDK 直接吃 n=50 的 2500 维 QUBO，哪怕 SDK 能跑——这是"凑数据"行为，违反铁律 §二.1。
   - **分解策略一致**：Q3 (n=50)、Q4 多车版必须先做大规模分解（聚类 / 扇区 / 滚动窗口），把每个子问题压到 ≤23² = 529 比特；SDK 与 CIM 必须用同一套分解 + 同一套子 QUBO，仅替换求解器后端。
   - **8-bit 精度预留**：SDK 阶段就用 `kw.qubo.adjust_qubo_matrix_precision(Q, bit_width=8)` 把矩阵压成 8-bit 整数；这样最后切到 CIM 真机不需要重新调参。
   - **代码模式**：脚本里写一个 `solve_subqubo(Q_sub, backend="sdk_sa" | "cim")` 抽象层，**只换一行**就能从 SDK 切到 CIM。
   - **诚信叙述**：论文里写 SDK 验证时**必须**说明该方案的子 QUBO 比特数 ≤ 550，并在结果表里列出"SDK SA 后端 / CIM 真机后端"两栏对比，禁止只展示 SDK 数据假装"已为真机准备好"。
7. **CIM 真机平台上传：矩阵维度必须 ≤ 550 比特**（实操硬约束）：玻色 CPQC-550 真机最多支持 550 个相干光振荡器，**手动上传 platform.qboson.com 时矩阵 ising 维度只能 ≤ 550**（不能超过；可以小于；**不需要 padding 到 550**）。具体要求：
   - **比特上限**：上传的矩阵 ising 维度（含辅助自旋）≤ 550；对应 QUBO 维度 ≤ 549。这与铁律 §二.6 一致，是同一硬约束的实操形式。
   - **可以直接传小矩阵**：Q4 的 49×49、Q1 的 225×225 子 QUBO 都可以**原样**上传，无需扩到 550×550；平台会自动接受任何 ≤ 550 的尺寸。
   - **8-bit 精度调整必须先做**：上传前必须 `kw.qubo.adjust_qubo_matrix_precision(Q, bit_width=8)` 把矩阵转成整数（CPQC-550 硬件位宽要求）。
   - **CSV 输出位置**：标准输出落到 `results/基础模型/qubo_matrices/csv_8bit/<name>_dense.csv`（8-bit 整数 dense 矩阵，n×n，n ≤ 549）。这就是平台手动上传的标准格式。
   - **配额警示**：平台每次 task 占 1 次配额（**不论矩阵实际维度大小**），所以分解越细 = 配额消耗越多——Q4 七车独立子 QUBO 要 7 次配额，Q3 三段要 3 次配额，Q1 + Q2 各 1 次，**全 4 问完整真机验证至少 12 次配额**。当配额不足时，优先选 **Q1 + Q3-seg1 + Q4-v3** 作为论文最低必跑集（覆盖单车 / 大规模分解 / 多车三种结构各一例）。
   - **解码取原 nvar**：从平台下载的 spin 解长度 = nvar + 1（含辅助自旋），取 spin[:nvar] 解码，最后 1 位丢弃。
   - **诚信叙述**：论文中说明"本文所有 QUBO 子问题的 ising 维度均 ≤ 550，可直接上传 CIM CPQC-550 真机求解"，禁止虚构"已上传到真机但实际未跑"的描述。
   - **真机结果归位约定**：所有 CIM 真机的求解输出（spin 解 zip、解码后 JSON、哈密顿量曲线图、与基线的对比图、平台返回的原始日志）一律落到 **`results/真机结果/`** 目录，**不要混入** `results/基础模型/` 或 `results/改进模型/`。子目录可按问题分组（如 `results/真机结果/q1/`、`results/真机结果/q2/` 等）。这一约定让"真机数据"与"经典/SDK SA 数据"在文件系统层物理隔离，避免论文里张冠李戴；同时方便答辩时被问"哪些数据来自真机"时秒级定位。

## 三、算法分层策略

- **递进结构**：基础模型 → 加约束 → 扩规模 → 多目标综合。例：TSP → 带时间窗 VRP → 大规模可分解 → 多车容量 VRP。
- **小规模**：精确算法（DP/枚举剪枝/整数规划/精确求解器），作为后续算法的正确性基线。
- **大规模**：可分解/混合算法。必须说明：为何精确不可行、为何完整 QUBO 不可行、初始解怎么来、局部如何搜索、稳定性如何判断。
- **创新点要真实**：来自抽象更整齐、变量更简洁、约束更干净、分层更自然、验证更充分。禁用"首创""国际领先"。
- **可复现**：每个算法交代输入、决策变量、目标函数、约束、关键参数（初始温度/降温系数/最大迭代/邻域操作/接受准则/随机种子）、终止条件、输出、代码位置。

## 四、写作语句模板

- **问题分析**：交代题目本质 + 与上一问的扩展关系。
- **模型建立**："设…为决策变量。以…为目标函数。考虑…约束，建立如下条件。最终得到…模型。"
- **算法选择**："由于…规模较小，采用…精确算法；规模扩大到…时直接求解会…，故采用…混合算法。优势在…，不足在…，后续通过…检验。"
- **结果分析**：每个关键数值后加 2–4 句——说明什么、是否符合直觉、与约束什么关系、对实际什么意义、有无异常。
- **结论**：逐题对应——"针对问题一，得到…；针对问题二，得到…；…综合来看，本文模型说明…"
- **摘要**：每问的方法 + 核心结果 + 关键数值 + 模型优势 + 是否检验，全文浓缩，不写背景。

## 五、必做检验

参数扰动 ±5/10/20%；不同算法对比；资源/权重变化；误差与可行性检查；稳定性与复杂度分析。**没有灵敏度分析的国奖论文几乎不存在。**

5. **产出文件清单**（**这一项最重要，写论文取图就靠这个表**）：

   - 图：`figures/fig_<编号>_<说明>.png` + `.pdf` + 一句话说明（图作用 + 论文里用在哪节）
6. **结论 1-2 句**：本次实验的发现/异常/对论文的价值（成功/失败都写，失败写为什么失败、下一步怎么改）；

**与素材库的关系**：实验日志是流水账（按时间），素材库是按章节归档（按内容）。每次写完日志条目，**判断是否有论文级阐述要顺手沉到素材库**——如果有，在日志条目末尾打勾"已同步"+ 索引；如果只是过程性记录则注明"暂不入素材库"。

## 十一、实验结果可视化（强制出图约定）

**核心原则**：**任何一次跑出数值结果的实验，都必须配图**——纯文字数值的实验报告进不了国奖论文。脚本结束前必须把所有该出的图都画出来落盘到 `figures/`，并在实验日志的"产出文件清单"里登记说明。

**图类型选择速查**（按结果性质选合适的图，不要乱配）：

| 结果性质 | 推荐图类型 | 必备元素 |
|---|---|---|
| **路径/路线方案** | 环形路径图 / 节点-边图 / 地理坐标散点连线 | 节点编号、起止 depot 高亮、箭头方向、违反节点配色（红=违反 橙=正常 灰=depot） |
| **时间维度调度** | 甘特图 / 时间窗 vs 实际到达条形图 | 时间窗背景框、实际服务区段、早到/晚到标注、横轴时间刻度 |
| **求解收敛过程** | 折线图（cost / energy / J vs 迭代/温度） | 横轴迭代步、纵轴指标、对数轴可选、最佳值标注 |
| **真机/SDK 哈密顿量演化** | 折线图（H vs time） | 题目硬性要求图，必须出 |
| **参数灵敏度（1 维）** | 折线图带误差棒 | 多 seed 平均 + std；x 轴参数、y 轴目标 |
| **参数灵敏度（2 维网格）** | 热图 (heatmap) | 双轴参数、cell 标数值、配色条；冷-热配色（viridis/RdYlGn） |
| **多算法对比** | 柱状图 / 箱线图 | 算法名横轴、目标纵轴、误差棒或多次运行分布 |
| **可行率/成功率** | 热图 / 柱状图百分比 | y 轴 0-100%、cell 标百分比 |
| **解的分布（多次运行）** | 散点图 / 直方图 / 箱线图 | x 轴 cost、y 轴 frequency 或种子 |
| **能量/解空间结构** | QUBO 矩阵热图 / 能级谱 | 矩阵尺寸、配色对称、零值用白色 |
| **多目标 trade-off** | Pareto 前沿散点图 | 双目标坐标、Pareto 点高亮、支配关系 |
| **算法消融** | 分组柱状图 / 雷达图 | 配置名、各指标对比 |
| **规模性 scalability** | 折线图 (n vs time / cost)，双轴或对数轴 | n 横轴；time 与 cost 各一线 |
| **节点属性分布** | 散点 + 时间窗矩形条 / 直方图 | 节点 ID、a_i/b_i 区间、需求量 |

**强制约定**：
1. **每次跑实验至少 2 张图**：① 解可视化（路径/调度/最终方案）+ ② 过程可视化（收敛/灵敏度/对比）。只有最终数值没有过程图的实验不合格。
2. **图必须落盘到 `figures/` 双格式**：`fig_<编号>_<说明>.png`（dpi≥300）+ `.pdf`（论文嵌入用矢量）。
3. **图必须在实验日志里有一句话说明**：图作用 + 论文哪一节用——例如"路径环形图（论文 §问题二·结果 4.1 节插图）"。
4. **图必须有清晰元素**：标题（含关键数值如 travel=31, J=84121）、轴标签带单位、图例、配色含义注释；中文用 `Microsoft YaHei` / `SimHei` 防乱码。
5. **不出图的脚本不算跑完**：哪怕只是 print 数值的脚本，也要把数值用 matplotlib 画出来再保存——这是论文级标准。
6. **失败实验也要出图**：失败/异常结果用图展示出来比纯文字日志更有论文价值（如可行率 0% 的热图比"失败"两字更说明问题）。

**图编号规则**：`fig_<问题编号 0X>_<功能简写>.png`。例：
- `fig_01_q1_route.png` — 问题一路径
- `fig_01_q1_sa_convergence.png` — 问题一 SA 收敛
- `fig_q1_grid_feasibility.png` — 问题一 A×T0 可行率热图
- `fig_02_q2_route.png` — 问题二路径
- `fig_02_q2_convergence.png` — 问题二 SA 收敛

**反例（禁止）**：
- 跑了脚本只 print 数字，不出图（"我先看数值合不合理"——错，看图比看数字快）；
- 出了图但没在实验日志登记（写论文时找不到）；
- 出了图但没标题/没轴标签/没单位（图自己讲不清自己）；
- 同一种图反复出（每个实验都只出"路径环形图"——其它过程信息哪儿去了）。
