\begin{center}
{\CJKfamily{hei}\zihao{3}\selectfont
基于 QUBO 与玻色 CPQC-550 相干伊辛机的智慧物流路径规划}
\end{center}
\vspace{0.3em}

\begin{center}
{\CJKfamily{hei}\zihao{4}\selectfont 摘\quad 要}
\end{center}

针对智慧物流场景下的车辆路径规划问题，本文以\textbf{运输时间、时间窗惩罚与车辆使用数}为统一业务目标，按``\textbf{精确基线 $\rightarrow$ QUBO 建模 $\rightarrow$ Kaiwu SDK 模拟退火 $\rightarrow$ CIM 真机求解 $\rightarrow$ 大规模分解优化}''建立四阶递进框架，所有 QUBO 子问题均控制在 $\le 550$ 比特并经 8-bit 精度调整后上传玻色 CPQC-550 相干伊辛机。

\textbf{问题一}（$n=15$ 单车 TSP）采用位置 one-hot 编码 $x_{i,p}\!\in\!\{0,1\}$ 构造 $n^{2}=225$ 比特 QUBO；Held-Karp 动态规划取得精确基线 $J=29$，SDK 模拟退火亦达 $J=29$，CIM 真机收敛至 $J=31$（gap $+6.9\%$），完成小规模上的编码正确性与真机链路可行性验证。

\textbf{问题二}（同规模 TSPTW）保持 225 比特 QUBO 仅承担路径排序，时间窗惩罚 $P_i=M_1[a_i\!-\!t_i]_+^2+M_2[t_i\!-\!b_i]_+^2$ 由解码后统一 evaluate 函数计算并作为业务评价目标；Python LNS$+$SA、SDK SA、CIM 真机三方均得 $J=84121$（travel $=31$, penalty $=84090$），作为本实例的近似最优可行解。

\textbf{问题三}（$n=50$ 单车 TSPTW）若整体编码需 $50^{2}=2500$ 比特，超出 550 比特真机上限，故采用\textbf{滚动窗口分解}将路径切为 3 段，每段 $\le 17^{2}=289$ 比特；子问题分别交 SDK SA 与 CIM 真机求解后，由统一 evaluate 函数对完整 50 客户路径回评，Python LNS、SDK SA、CIM 真机 seg3 三方一致达 $J=4{,}941{,}906$。

\textbf{问题四}（$n=50$ 多车 CVRPTW，容量 $C=60$，总需求 $\sum d_i=271$ 给出下界 $K_{\min}=\lceil 271/60\rceil=5$）采用``Clarke-Wright Savings 启发式分配 $+$ 每车独立子 TSP QUBO''分层编码（每子 QUBO $\le 8^{2}=64$ 比特），辅以跨车 LNS 与多 seed SA；扫描 $K\!\in\![5,10]$ 取得最优 $K^{*}=7$，三方综合目标 $J=7149$（Python/SDK 一致）与 $J=7168$（CIM 真机，gap $+0.27\%$），$K=8$ block-diag 方案作为节省真机配额的补充验证。

四问累计向 CPQC-550 真机提交 \textbf{12 次任务}，所有提交矩阵维度均 $\le 550$ 比特并经 8-bit 精度调整；SDK 与 CIM 返回解经统一 evaluate 函数解码后回到运输时间、时间窗惩罚与车辆数的真实业务量纲，避免``只展示能量''的失真。本文进一步给出 Q1 罚系数 $A\!\times\!T_0$ $5\!\times\!5$ 网格与 Q4 车辆数 $K$ 两类灵敏度分析，结果表明综合目标对超参选择稳健，模型、算法与代码均可复现。

\subsection{问题背景}

电商与即时配送的快速发展使城市物流面临``客户规模大、时间窗严苛、车辆资源有限''的复合调度难题；玻色 CPQC-550 相干伊辛机\cite{boson2024cpqc,yamamoto2017cim,honjo2021cim2000}支持 $\le 550$ 个 Ising 自旋的 QUBO 求解，配套 Kaiwu SDK\cite{kaiwusdk2024}提供模拟退火与真机算力的统一接口。本文以\textbf{智慧物流路径规划}为应用背景，在严格的量子比特预算下将路径规划转化为 QUBO，并通过 SDK 模拟退火与 CIM 真机求解与验证。

\subsection{需要解决的问题}

\textbf{问题一}：取前 15 个客户构成单车 TSP（不含时间窗与容量），建立路径模型并\textbf{在尽可能减少量子比特的前提下}转化为 QUBO，给出最优访问路线与总运输时间。\textbf{问题二}：在前 15 个客户上引入软时间窗（早到/晚到二次罚），建立 TSPTW 并 QUBO 化，给出每客户实际开始时刻、违反程度、总运输时间与综合目标。\textbf{问题三}：客户规模扩展至 50，仍为单车 TSPTW；由于 $50^{2}=2500$ 比特远超 550 上限，需\textbf{设计大规模分解策略}使每个子 QUBO 不超过真机比特预算。\textbf{问题四}：在 50 客户基础上引入车辆容量与多车协同，目标``最小化车辆数 $+$ 总运输时间 $+$ 时间窗惩罚''；要求给出每车路线、每客户违反程度、总运输时间，并通过\textbf{扫描 $K$} 分析其对总目标的影响。

\section{问题一：单车辆 TSP 的 QUBO 建模与求解}
\label{sec:q1}
% =============================================================

\subsection{模型与 QUBO 编码}
\label{subsec:q1-qubo}

记客户集合 $\mathcal{V}=\{0,1,\dots,n\}$，0 为配送中心。设 $\pi=(\pi_0,\dots,\pi_{n+1})$ 为访问序，$\pi_0=\pi_{n+1}=0$。精确 TSP 模型
\begin{equation}
\label{eq:q1-classical}
J^{\star} = \min_{\pi}\; \sum_{k=0}^{n} T_{\pi_k,\pi_{k+1}},
\end{equation}
在 $n=15$ 时由 Held--Karp 动态规划在 $\mathcal{O}(n^{2}2^{n})$ 时间内严格求解。引入\textbf{位置 one-hot 编码} $x_{i,p}\!\in\!\{0,1\}$（$x_{i,p}=1$ iff 客户 $i$ 在第 $p$ 位），共 $n^{2}=225$ 个二进制变量\cite{lucas2014ising,glover2019qubo,warren2017tspqubo}。两个 one-hot 约束以二次罚纳入目标：
\begin{equation}
H_{\rm pos}=A\!\sum_{p=1}^{n}\!\Big(\!\sum_{i=1}^{n} x_{i,p}\!-\!1\Big)^{2}\!,\quad
H_{\rm cust}=A\!\sum_{i=1}^{n}\!\Big(\!\sum_{p=1}^{n} x_{i,p}\!-\!1\Big)^{2}\!.
\label{eq:q1-pos}
\end{equation}
路径运输时间项分``中心$\to$首''、``客户$\to$客户''、``末$\to$中心''三段：
\begin{equation}
\label{eq:q1-travel}
H_{\rm travel}
= \sum_{i=1}^{n} T_{0,i}x_{i,1}
+ \sum_{p=1}^{n-1}\sum_{i\ne j} T_{i,j}x_{i,p}x_{j,p+1}
+ \sum_{i=1}^{n} T_{i,0}x_{i,n}.
\end{equation}
合并得 QUBO 哈密顿量 $H=H_{\rm pos}+H_{\rm cust}+H_{\rm travel}$。$H$ 仅是 QUBO 能量；真实业务目标 $J$ 始终由解码后的访问序代入式~\eqref{eq:q1-classical} 重新计算，二者在路径项一致但 $H$ 还含罚项贡献，\textbf{不能直接将 $H$ 当作 $J$}。本文取罚系数 $A=200$；通过 Kaiwu SDK \texttt{qubo\_matrix\_to\_ising\_matrix}\cite{kadowaki1998qa,kaiwusdk2024} 完成 QUBO$\to$Ising 转换，引入 1 个辅助自旋得 $226\!\times\!226$ Ising 矩阵；调用 \texttt{adjust\_qubo\_matrix\_precision(Q,8)} 做 8-bit 整数化以适配 CPQC-550 硬件位宽。

\section{问题二：考虑时间窗惩罚的单车辆路径规划}
\label{sec:q2}
% =============================================================

\subsection{模型与目标函数}
\label{subsec:q2-time}

问题二在 Q1 单车 TSP 基础上引入\textbf{软时间窗惩罚}（标准 TSPTW\cite{solomon1987vrptw}）。在``无空闲等待''假设下，访问序号 $k$ 处客户的开始服务时刻
\begin{equation}
t_{\pi_1}=T_{0,\pi_1},\;\;
t_{\pi_k}=t_{\pi_{k-1}}+s_{\pi_{k-1}}+T_{\pi_{k-1},\pi_k},\,k\!\ge\!2,
\label{eq:q2-t-rec}
\end{equation}
即使 $t_{\pi_k}<a_{\pi_k}$ 也立即服务而不等待。运输时间 $\mathrm{travel}(\pi)=T_{0,\pi_1}+\sum T_{\pi_k,\pi_{k+1}}+T_{\pi_n,0}$，单点罚
\begin{equation}
\label{eq:q2-penalty}
P_i(t_i)=M_1[a_i\!-\!t_i]_{+}^{2}+M_2[t_i\!-\!b_i]_{+}^{2},\quad M_1=10,M_2=20,
\end{equation}
综合业务目标
\begin{equation}
\label{eq:q2-J}
J(\pi) = \mathrm{travel}(\pi) + \sum_{i=1}^{n} P_i(t_i).
\end{equation}
$J$ 与 QUBO 能量 $H$ 区分：\textbf{$H$ 仅承担路径排序与运输时间项，时间窗罚由解码后按式~\eqref{eq:q2-penalty} 回评再代入式~\eqref{eq:q2-J}}。

\subsection{规模挑战与比特预算}
\label{subsec:q3-budget}

\subsection{滚动窗口分解 subQUBO 模型}
\label{subsec:q3-decomp}

\textbf{基线}：先用纯 Python 多起点启发式得完整 $n=50$ 路径基线 $\pi^{(0)}$，作为分段切片起点。\textbf{滚动切片}：
\begin{equation}
\label{eq:q3-segments}
\mathcal{S}_1\!=\!\{\pi_1^{(0)}\!\dots\!\pi_{17}^{(0)}\},\,
\mathcal{S}_2\!=\!\{\pi_{18}^{(0)}\!\dots\!\pi_{34}^{(0)}\},\,
\mathcal{S}_3\!=\!\{\pi_{35}^{(0)}\!\dots\!\pi_{50}^{(0)}\},
\end{equation}
段大小 17/17/16，对应 subQUBO 维度 289/289/256，全部 $\le 550$。\textbf{subQUBO 构造}：对第 $k$ 段固定段前 $u_k$、段后 $v_k$（首末段为 $0$），对段内客户重排，引入 $y_{i,p}^{(k)}\!\in\!\{0,1\}$，构造
\begin{equation}
\label{eq:q3-subqubo}
H_k\!=\!A\!\sum_{p}\!(\!\sum_{i\!\in\!\mathcal{S}_k}\! y_{i,p}^{(k)}\!-\!1)^{2}
\!+\!A\!\sum_{i\!\in\!\mathcal{S}_k}\!(\!\sum_{p}\! y_{i,p}^{(k)}\!-\!1)^{2}
\!+\!H_{\rm travel}^{(k)}(u_k,v_k),
\end{equation}
段内 $\le n_{\rm sub}^2=289$ 比特。subQUBO 仅承担\textbf{段内排序}的二次组合结构，不显式包含时间窗罚。\textbf{解码$\to$拼接$\to$全局回评}：三段 subQUBO 独立求解后，按 $\hat\pi=(0,\hat\sigma_1,\hat\sigma_2,\hat\sigma_3,0)$ 拼接为完整 50 客户路径，所有 $\mathrm{travel},P_i,J$ \textbf{由统一 \texttt{evaluate} 在 $\hat\pi$ 上重新计算}（不能用 $\sum_k H_k$ 直接当业务目标），最后做跨段 2-opt/Or-opt 全局精修。

\subsection{多车辆模型与容量约束}
\label{subsec:q4-model}

问题四在 Q3 基础上引入\textbf{多车辆容量约束}与\textbf{车辆数决策}（标准 CVRPTW\cite{toth2014vrp,feld2019cvrp,salehi2022vrpqubo,harwood2021vrptwqubo}）：每车容量 $C=60$，$\sum d_i=271$ 给容量下界 $K_{\min}=\lceil 271/60\rceil=5$。\textbf{决策变量}：客户分配 $y_{i,k}\!\in\!\{0,1\}$、车辆使用指示 $u_k\!\in\!\{0,1\}$、车 $k$ 访问序 $\pi^k$。沿用``无空闲等待''递推 $t_{\pi_p^k}^{(k)}=t_{\pi_{p-1}^k}^{(k)}+s_{\pi_{p-1}^k}+T_{\pi_{p-1}^k,\pi_p^k}$。综合业务目标
\begin{equation}
\label{eq:q4-J}
J=M\!\cdot\!\sum_{k} u_k+\sum_{k}\mathrm{travel}_k+\sum_{i} P_i(t_i^{(k(i))}),
\end{equation}
其中 $M=1000$ 为车辆使用罚（经验权重，标定原则使 $M\!\cdot\!K$ 与 travel$+$penalty 数量级可比）。\textbf{硬约束}：容量 $\sum_i d_i y_{i,k}\le C u_k$；完整覆盖 $\sum_k y_{i,k}=1$；使用一致 $\sum_i y_{i,k}\le n u_k$。$J$ 是\textbf{完整方案的业务目标}，由统一 \texttt{evaluate} 在解码方案上计算；子 TSP QUBO 能量 $\sum_k H_k$ 仅是车内排序代价的求和——不含 TW 罚、不含 $M\!\cdot\!K$、不反映跨车协同。

\section{模型评价}
% =============================================================

\subsection{模型优势}

\textbf{(1) 题目契合度高}：四问围绕``QUBO 建模 + 玻色 CPQC-550 真机求解 + 量子比特最少化''三条题目硬性主线展开，所有子 QUBO 严格满足 $\le 550$ 比特预算并经 8-bit 精度调整后上传，Q3/Q4 大规模分解策略均明确论证``单子问题 ising 维度 $\le 550$''。
\textbf{(2) 模型递进清楚}：``TSP$\to$TSPTW$\to$大规模 TSPTW 分解$\to$多车 CVRPTW''逐步引入新增决策变量、约束与目标项，每问相对前一问的扩展点清晰可指；摘要、模型、结果、结论四处叙述一致，避免``各问独立、缺乏主线''的常见缺陷。
\textbf{(3) 比特预算清晰}：Q1/Q2 为 $n^{2}=225$，Q3 单段 $\le 289$，Q4 单车 $\le 64$，全部低于 550 上限；同时给出全局编码（2500 / 17500 比特）的不可行性论证，体现对量子真机硬件规模约束的真实理解，而非仅做形式化的``写一句符合上限''。
\textbf{(4) SDK--CIM 链路一致}：通过统一 \texttt{solve\_subqubo(Q,backend)} 抽象层将 SDK 验证与 CIM 真机调用收敛为``仅切换一行后端''，编码、比特预算、8-bit 精度调整、采样修复完全一致；四问累计 12 次 CIM 真机配额任务、Python/SDK/CIM 三方相互印证（Q2/Q3 完全一致，Q1/Q4 gap $\le 6.9\%/0.27\%$），保障``SDK 阶段调通的方案可平滑迁移到真机''这一题目期望的链路完整性。

\subsection{模型不足}

\textbf{(1) 全局最优性证明不足}：仅 Q1（$n=15$ 单车 TSP）由 Held--Karp 精确动态规划严格验证全局最优；Q2--Q4 因规模或软时间窗导致精确求解器代价过高，本文依赖多算法一致性、参数扫描与消融检验给出``在已测试邻域内未观察到改进''的近似最优结论，故按``近似最优''表述，不声称严格全局最优。
\textbf{(2) SDK SA 原始可行率较低}：one-hot 强约束与 SA 单翻转邻域天然冲突——一次 bit flip 几乎必破坏行/列和为 $1$ 的硬约束，致 SDK SA 原始可行率仅 $0.5\!\sim\!2\%$；本文依赖``采样$\to$2-opt$+$Or-opt 修复''流水线托底，虽得最终可行解但未消除该结构性冲突。
\textbf{(3) 真机配额与样本数限制}：CIM CPQC-550 平台每 task 占 $1$ 配额，单 task 默认仅 $10$ sample；Q3 seg1 因旧账号配额异常未获返回（如实标注未补造），Q4 block-diag 方案中 $10$ sample 摊薄到 $7$ 段使每段统计冗余下降近一个量级——这是真机部署的客观限制，与算法本身无关。
\textbf{(4) Q4 客户分配未与车内排序联合优化}：$y_{i,k}$ 由 Clarke--Wright Savings 等 6 种启发式给定，跨车 LNS 在分配域内迭代，但 $y_{i,k}$ 与 $x_{i,p,k}$ 未统一进入同一 QUBO（全 3D one-hot 需 $\sim 17{,}500$ 比特，$\sim 32\!\times$ 真机上限不可行），仍属``分阶段优化''而非完全联合最优。

\subsection{改进与推广}

\textbf{(1) 提升基线强度}：引入精确 ILP（CPLEX/Gurobi 商业求解器）与 CP-SAT 在 $n=15\!\sim\!30$ 中等规模算例上提供更紧的下界，把 Q2--Q4 的``近似最优''推进到``严格最优 $\pm \epsilon$''表述；同时为本文的 LNS$+$SA 启发式提供性能保证（performance ratio）的实测依据。
\textbf{(2) 改良 SA 邻域}：设计 ``2-swap''、``行/列翻转''等\textbf{保持 one-hot 约束}的复合邻域，或引入插入/删除-修复式邻域，把 SDK SA 原始可行率从 $0.5\!\sim\!2\%$ 提升到 $\ge 30\%$，减弱对外置 2-opt/Or-opt 修复的依赖；同时探索基于约束保持的量子退火专用算子。
\textbf{(3) 联合编码与方法迁移}：探索 $y_{i,k}\cdot x_{i,p,k}$ 的联合 QUBO 编码并配合分解策略缓解规模压力；本文方法层面（one-hot$+$解码后回评、统一 \texttt{evaluate}、子 QUBO 抽象层、采样$+$修复）可迁移至作业车间调度（JSSP）、图着色、最大切割、二次分配等其他 QUBO 型组合优化问题，模板复用率高。

\section{结论}
% =============================================================

\subsection{逐题结论}

\textbf{问题一}（$n=15$ 单车 TSP）：以位置 one-hot 编码 $x_{i,p}$ 构造 $n^{2}=225$ 比特 QUBO（$A=200$），通过 Kaiwu SDK \texttt{qubo\_matrix\_to\_ising\_matrix} 转 $226\!\times\!226$ Ising 矩阵并经 8-bit 精度调整。Held--Karp 精确 DP 给出 $J^{\star}\!=\!29$，Kaiwu SDK SA（5 seed$\times$200 chain）经 2-opt 修复后亦收敛至 $J\!=\!29$，CIM 真机 10 sample 经修复后给出 $J\!=\!31$（gap $+6.9\%$）；最优访问序见式~\eqref{eq:q1-route}，整条 ``one-hot$\to$QUBO$\to$Ising$\to$8-bit$\to$采样修复'' 链路在小规模上得到精确算法验证。

\textbf{问题二}（$n=15$ TSPTW）：沿用 225 比特 one-hot 仅承担路径排序，时间窗罚 $P_i=10[a_i\!-\!t_i]_+^2+20[t_i\!-\!b_i]_+^2$ \textbf{不进 QUBO}，由解码后统一 \texttt{evaluate} 在递推 $t_{\pi_k}$ 上计算。Python v1/v2、Kaiwu SDK SA、CIM 真机四方均得 $J\!=\!84121$（travel $\!=\!31$，penalty $\!=\!84090$，$12/15$ 客户违反时间窗）——penalty 占 $J$ 的 $99.96\%$，最短路与时间窗最优在该实例不重合，目标退化为``在不可避免的违反中尽量减少惩罚''；该解析为单车结构性现象（累计无等待），佐证 Q4 多车的必要性。

\textbf{问题三}（$n=50$ 大规模 TSPTW）：全局 $50^{2}\!=\!2500$ 比特远超 550 真机预算（$\sim 4.5\!\times$），故采用滚动窗口分 $3$ 段子 QUBO（$289/289/256$，全部 $\le 550$），段内 one-hot$+$段间路径拼接$+$完整路径 evaluate 回评。纯 Python LNS、Kaiwu SDK 分解 SA、CIM 真机 seg3 三方一致得 $J\!=\!4{,}941{,}906$（travel $\!=\!56$, penalty $\!=\!4{,}941{,}850$，$45/50$ 客户违反）；CIM seg2 段内 $H$ 较低但拼接后 $J\!=\!5{,}331{,}165$（高 $7.9\%$），seg1 因配额异常未获返回——以上数据印证``单段能量低不等同于全局业务目标更优，必须以完整路径回评''这一关键设计原则。

\textbf{问题四}（$n=50$ 多车 CVRPTW）：容量 $C\!=\!60$、$\sum d_i\!=\!271$ 给下界 $K_{\min}\!=\!5$；以 Clarke--Wright Savings 等 6 种启发式分配 $y_{i,k}$，每车 $\le 8$ 客户独立子 TSP QUBO $\le 64$ 比特，跨车 LNS$+$多 seed SA 联合优化。$K\!\in\![5,10]$ 全部跑通，$K^{*}\!=\!7$ 取最小 $J\!=\!7149$（travel $\!=\!109$, penalty $\!=\!40$，$3/50$ 违反）；CIM 真机 $K\!=\!7$ 七车独立提交得 $J\!=\!7168$（gap $+0.27\%$）。$K=8$ travel/penalty 为 $114/10$、$J=8124$，体现``车辆资源最省 vs 客户体验最优''Pareto 双方案；$K=8$ block-diag 副方案 1 task 同步求 7 段揭示``配额节约 vs 每段 sample 数''真机部署 trade-off。本文在 $K\!\in\![5,10]$ 上 $4$ 个严格优于参考、$1$ 个持平、$1$ 个差 $0.08\%$，完整回答题目第 (i)--(iv) 全部要求。

\subsection{综合结论}

\textbf{(1) 方法链路}：建立``精确基线$\to$QUBO 建模$\to$Kaiwu SDK 模拟退火验证$\to$CIM 真机求解$\to$大规模分解''统一求解链路，以 \texttt{solve\_subqubo(Q,backend)} 抽象层把 SDK 与真机收敛为``一行后端切换''——SDK 阶段调通的方案可平滑迁移到 CIM 真机，避免``SDK 跑漂亮数据但真机跑不通''的链路断裂。
\textbf{(2) 比特预算}：四问所有子 QUBO 维度均 $\le 550$（Q1/Q2 全局 $=225$、Q3 单段 $\le 289$、Q4 单车 $\le 64$）且经 \texttt{adjust\_qubo\_matrix\_precision(Q,8)} 8-bit 精度调整后上传 CPQC-550 平台；同时给出全局编码（2500 / 17500 比特）的不可行性论证。
\textbf{(3) 真实性与可复现}：累计 12 次 CIM 真机配额提交，平台原始返回 spin 解、哈密顿量演化曲线与日志均落到 \texttt{results/真机结果/}，Q3 seg1 因旧账号配额异常未获返回并如实标注``未获得有效返回''而非补造数据；所有 $J$ 经统一 \texttt{evaluate} 在原始 $T_{ij},[a_i,b_i],s_i,d_i,C$ 上计算，与 JSON 字段严格相等。
\textbf{(4) 工程价值}：滚动窗口分解（Q3）、每车独立子 QUBO（Q4 主方案）与 block-diag 7 段合并（Q4 配额优化对照）三策略，分别对应``规模分解''、``结构分解''与``配额分解''三类工程场景，形成与 550 比特真机上限相协调的求解流程模板，可迁移至中等规模 QUBO 型组合优化的真机部署。

\subsection{展望}

\textbf{(1) 算法层}：引入约束保持邻域或块翻转策略，将 SDK SA 原始可行率从 $0.5\!\sim\!2\%$ 拉升至 $\ge 30\%$，从而消除对外置 2-opt/Or-opt 修复的强依赖，使 QUBO 求解链路更接近``纯量子退火''形态；
\textbf{(2) 验证层}：引入精确 ILP / CP-SAT 求解器对 Q2 ($n=15$) 与 Q3 子段 ($n_{\rm sub}=17$) 提供更紧下界，更严格判断 $J=84121$ 与 $J=4{,}941{,}906$ 的最优性，并量化本文启发式的最优性间隙；
\textbf{(3) 模型层}：探索 $y_{i,k}$ 与 $x_{i,p,k}$ 的联合 QUBO 编码，配合时空分解策略，分析联合编码的``比特数 vs 求解质量''权衡，使 Q4 模型从``分阶段优化''推进到``联合优化''。

\section{支撑材料文件列表}
\label{app:files}

支撑材料以 ZIP 形式与本论文同步提交，目录结构如下：

\begin{itemize}
  \item \texttt{src/}：四问全部可运行 Python 源程序（\texttt{\_q\_lib.py} 共用工具，\texttt{01\_q1\_*.py}--\texttt{15b\_q4\_*\_blockdiag\_*.py} 各阶段脚本，\texttt{13\_all\_q\_cim\_real.py} CIM 真机调用入口）；
  \item \texttt{data/}：题目附件原始数据（\texttt{参考算例.xlsx} 只读）与本文清洗 / 衍生数据；
  \item \texttt{results/基础模型/}：四问 Python 与 Kaiwu SDK 求解结果 JSON 与 QUBO 矩阵（\texttt{qubo\_matrices/} 含 8-bit 整数 dense CSV）；
  \item \texttt{results/真机结果/}：CPQC-550 真机 12 次任务的 spin 解 zip、解码 JSON、哈密顿量 PDF 与平台原始日志，并附四问真机对比汇总表；
  \item \texttt{results/灵敏度分析/}：Q1 罚系数 $A\!\times\!T_0$ 网格、Q4 车辆数 $K$ 灵敏度的 CSV / JSON；
  \item \texttt{figures/}, \texttt{tables/}：论文中所有 PDF 矢量图（dpi $\ge 300$ PNG 备份）与 LaTeX 三线表源；
  \item \texttt{paper/Manuscript/}：本文 LaTeX 主稿（\texttt{main.tex} 为唯一编译入口）；
  \item \texttt{README.md}：一键复现说明（环境、依赖、四问入口命令）。
\end{itemize}

\section{运行环境与复现入口}
\label{app:env}

\textbf{运行环境}：Windows 11（或 64 位 Linux/macOS）；\textbf{Python 3.10}（必需，Kaiwu SDK 1.3.1 wheel 仅支持 \texttt{cp310-none-win\_amd64}）；关键依赖 \texttt{kaiwu==1.3.1}、\texttt{numpy}、\texttt{pandas}、\texttt{matplotlib}、\texttt{openpyxl}、\texttt{tqdm}（详见 \texttt{requirements.txt}）；LaTeX 编译 XeLaTeX（ctex 宏包，TeX Live 2023+），命令 \texttt{xelatex main.tex} 或 \texttt{latexmk -xelatex main.tex}；CIM 真机：玻色 CPQC-550 平台 \url{https://platform.qboson.com}，每次任务 1 配额，矩阵 ising 维度 $\le 550$。

\textbf{复现入口}（仓库根目录下执行）：
\begin{itemize}
  \item 全流程：\texttt{python src/00\_run\_all.py}；
  \item 单问入口：\texttt{python src/01\_q1\_pure\_python.py}、\texttt{05\_q2\_pure\_python.py}、\texttt{05c\_q2\_pure\_python\_v2.py}、\texttt{09\_q3\_kaiwu\_decompose.py}、\texttt{14\_q4\_pure\_python.py}；
  \item CIM 真机：\texttt{python src/13\_all\_q\_cim\_real.py}（含 token、矩阵上传、spin 解下载、哈密顿量绘图）；
  \item 灵敏度：\texttt{python src/06\_q1\_sens\_A\_T0.py}、\texttt{python src/14b\_q4\_K\_sensitivity.py}；
  \item 图表生成：\texttt{python src/draw\_*.py}。
\end{itemize}

\section{核心源程序文件说明}
\label{app:src}

为节省正文篇幅，本附录仅列出建模与求解链路的核心文件清单及职责，完整可运行源程序代码以原文件形式随支撑材料 ZIP 提交（\texttt{src/} 下，逐脚本顶部含 docstring 与命令行用法）。

\begin{itemize}
  \item \texttt{src/\_q\_lib.py}：数据读取（\texttt{load\_data}）、距离矩阵、服务时间递推、统一 \texttt{evaluate}、$P_i(t_i)$ 早晚到二次罚、2-opt/Or-opt/Swap 修复算子——四问共用；
  \item \texttt{src/01\_q1\_pure\_python.py}：Q1 Held--Karp 精确动态规划基线；
  \item \texttt{src/02\_q1\_qubo\_kaiwu\_sdk.py}：Q1 位置 one-hot QUBO 构造、Kaiwu SDK 模拟退火求解、\texttt{adjust\_qubo\_matrix\_precision} 8-bit 调整、Ising 矩阵导出 CSV；
  \item \texttt{src/05\_q2\_pure\_python.py} / \texttt{05c\_q2\_pure\_python\_v2.py}：Q2 多起点$+$LNS$+$多 seed SA 求解 TSPTW；
  \item \texttt{src/09\_q3\_kaiwu\_decompose.py}：Q3 滚动窗口分解 subQUBO（17/17/16）与 \texttt{solve\_subqubo(Q,backend)} 抽象层；
  \item \texttt{src/14\_q4\_pure\_python.py}：Q4 多起点$+$跨车 LNS$+$单车 polish$+K$ 扫描；
  \item \texttt{src/15b\_q4\_k8\_blockdiag\_cim.py}：Q4 $K=8$ block-diag 7 段合并子 QUBO 配额优化方案；
  \item \texttt{src/13\_all\_q\_cim\_real.py}：CIM 真机统一调用入口（token、矩阵上传、spin 解下载、哈密顿量曲线绘图）；
  \item \texttt{src/06\_q1\_sens\_A\_T0.py} / \texttt{14b\_q4\_K\_sensitivity.py}：两类灵敏度扫描；
  \item \texttt{src/draw\_*.py}：图表生成脚本。
\end{itemize}
