## R-Q2-001 · 2026-04-26 · 纯 Python 多起点 polish + SA（含时间窗惩罚）

- **脚本**：[src/05_q2_pure_python.py](../../src/05_q2_pure_python.py)
- **命令**：`python src/05_q2_pure_python.py`
- **参数**：
  - n=15；M1=10，M2=20（时间窗早到/晚到二次惩罚系数）
  - 13 个 warm start：8 个随机 + NN(距离) + NN(综合分含时窗紧迫度) + sort by a_i / b_i / (a_i+b_i)/2
  - polish 流程：2-opt + Or-opt(1/2/3) + swap 三件套轮转直到 3 套都不再改进
  - SA：T0=80，Tf=1e-3，α=0.995，iter_per_T=400，邻域 swap/insert-segment/reverse 等概率
  - seed=20260426
- **核心数值结果**：
  - 路径：`0 → 2 → 13 → 6 → 5 → 8 → 7 → 11 → 10 → 1 → 9 → 3 → 12 → 4 → 15 → 14 → 0`
  - 总运输时间 travel = **31**（注：与 Q1 全局最优 cost=29 接近但更长 2 单位，因为 Q2 路径需要兼顾时间窗）
  - 时间窗惩罚 penalty = **84090**
  - 目标 J = travel + penalty = **84121**
  - 12/15 客户违反时间窗（绝大多数晚到，最严重客户 14 晚到 37 单位 → 罚 27380）
  - polish 用时 0.36s；SA 用时 20.54s（SA 未在 polish 解上进一步改进）
  - TOP-3 polish J: [84121, 86251, 88359]，差距小说明局部最优空间狭窄
- **产出文件**：
  - JSON：`results/基础模型/q2_pure_python.json` — 完整路径 + 逐客户 schedule + topk_polish_J
  - 表：`tables/tab_02_q2_schedule.csv` + `.tex` — 逐客户到达/早到/晚到/惩罚/离开（论文 §问题二·结果章节正文表）
  - 图：`figures/fig_02_q2_route.png` + `.pdf` — (a) 环形路径图（红=违反/橙=正常/灰=depot）+ (b) 时间窗 vs 实际到达甘特（绿=未违反/红=违反）
  - 图：`figures/fig_02_q2_convergence.png` + `.pdf` — SA 求解收敛曲线（外层降温步 vs 最佳 J）
- **结论 / 论文价值**：
  - 时间窗惩罚 penalty=84090 远超 travel=31（**罚值占目标 99.96%**），说明 n=15 的时间窗设置极其紧张：客户 b_i 多落于 [12, 30]，而走完 15 个点 + 服务时间需 ≥ 45 单位，**至少 12 个客户晚到不可避免**。
  - 该现象是 Q2 与 Q1 的核心差异：**Q1 是纯结构问题，Q2 已退化为"在不可避免的违反中尽量少违反"的最小化问题**。
  - 当前结果未做精确性验证（n=15 含时间窗的 DP 状态需 (mask, last, arrival)，可行但未跑）→ 标"近似最优解"，铁律 §二.1。
- **失败/异常**：无。SA 在 polish 解之上 20s 没改进，说明多起点 polish 已在该邻域结构下饱和，需要更强邻域（如 4-opt / large neighborhood search / 路径段交换）才可能突破。
- **是否同步素材库**：**否**——素材库 `00_论文素材库_问题二.md` 尚未建。下一次跑完 Kaiwu SDK 验证（Q2 + 时间窗 QUBO）后一并建立并同步。

## R-Q2-002 · 2026-04-26 · 纯 Python 强化求解器 v2（多起点 + LNS + 多 seed SA + 3-opt 精修）

- **脚本**：[src/06_q2_pure_python_v2.py](../../src/06_q2_pure_python_v2.py)
- **命令**：`PYTHONIOENCODING=utf-8 python -u src/06_q2_pure_python_v2.py`
- **相对 v1 的算法升级**：
  1) Warm starts：13 → **27 个**（16 随机 + 5 种 NN 综合分 + 5 种排序 + 罚意识贪心）
  2) **新增 LNS（大邻域搜索）**：150 轮 ruin-and-recreate；每轮按客户惩罚高低优先摧毁 3-6 个 + regret-2 重插 + polish；伴随 perturbation kick
  3) SA：单 seed → **多 seed (3 个)** 长退火（T0=200, α=0.995, iter/T=300）
  4) 邻域增加 3-opt-seg 切段重排
  5) 最终精修：在 polish/LNS/SA 三类候选最优解上跑一次重型 polish_full（含 3-opt 蛮力）
- **核心数值结果**：
  - 路径：`0 → 2 → 13 → 6 → 5 → 8 → 7 → 11 → 10 → 1 → 9 → 3 → 12 → 4 → 15 → 14 → 0`（**与 v1 完全一致**）
  - travel = 31，penalty = 84090，**J = 84121**
  - 与 v1 对比：ΔJ = 0，**改进 = 0%**
  - polish 用时 0.69s（27 个起点），LNS 1.66s，SA 53.42s，总 ≈ 56s
  - SA 3 个 seed 全部收敛到 J=84121（极强一致性）
  - LNS 150 轮也未跳出 84121
- **产出文件**：
  - JSON：`results/改进模型/q2_pure_python_v2.json` — 完整算法配置 + sa_pool 各 seed 结果 + v1 baseline 对比
  - 表：`tables/tab_02b_q2_v2_schedule.csv` + `.tex` — 最终调度表（与 v1 同解）
  - 图 1：`figures/fig_02b_q2_v2_route.png` + `.pdf` — 环形路径 + 时间窗甘特（**论文 §问题二·结果章节主插图**）
  - 图 2：`figures/fig_02b_q2_v2_compare.png` + `.pdf` — **v1 vs v2 J 与 penalty 柱状图对比**（论文 §问题二·算法对比节插图，证明算法饱和）
  - 图 3：`figures/fig_02b_q2_v2_convergence.png` + `.pdf` — 最优 seed SA 收敛曲线（α=0.995, iter/T=300）
  - 图 4：`figures/fig_02b_q2_v2_lns_history.png` + `.pdf` — LNS 大邻域搜索 best J 历史曲线（论文 §问题二·算法过程节插图）
- **结论 / 论文价值**：
  - **多种独立算法（27 个 warm start polish + 150 轮 LNS + 3 seeds 长 SA + 3-opt 蛮力）全部收敛到 J=84121**——这构成强证据：**该解是该邻域下的稳定局部最优（接近全局最优）**。
  - 可在论文中写：「为验证解的稳健性，本文以 27 种不同起点 + LNS 大邻域搜索 + 多 seed 长程 SA 三类异构算法独立求解，结果均收敛至同一 J = 84121，路径完全一致，可推断该解在算法可达邻域内已达饱和，铁律 §二.1 表述为'近似最优解'。」
  - **客户 14、15、12 的 b_i 远小于其在序列中的位置时刻**（窗 b 在 13-20 但被排到时刻 41-55），构成时间窗结构性紧张的最强证据。
  - 与 v1 的对比图（图 2）说明：**对该问题，更复杂的算法不一定带来更优解；v1 已饱和，差异主要体现在算法稳健性论证**。
- **失败/异常**：第一次跑了 6+ 分钟未结束（多起点 polish 含 3-opt + 8 seeds SA），违反"3 分钟内完成"，已 kill 重写：把 3-opt 移出 polish 循环只用于最终精修；SA 减到 3 seeds + α=0.995 + iter/T=300。最终 56s 完成。
- **是否同步素材库**：**否**（素材库 `00_论文素材库_问题二.md` 仍待建）。等做完 SDK 验证 + 灵敏度（M1, M2）后一并建立素材库并把这两次结果沉淀进 §4 实验结果章节。

### 主实验：方案 C（最终采用）

- **参数**：A_pen=200；SA baseline (T0=100, α=0.99, iter/T=10, size_limit=50, seed=20260426)；SA tuned (T0=500, α=0.995, iter/T=80, size_limit=200, 5 seeds × 200 chains, NN warm start)
- **关键改进**：**对 SDK pool 中所有可行解都做 polish**，按 J 取最优（不只对"QUBO 能量最优"那一个 polish）。SDK 找到的"QUBO 能量最优"路径（cost=29 附近的纯 TSP 最优）未必是 J 最优；polish 后 J 分布 = [84121, 86251, 86251, 88532, 88532, 88532]，最优落在 QUBO 能量序排第 2/3 的候选上。
- **核心数值结果**：
  - 路径：`0 → 2 → 13 → 6 → 5 → 8 → 7 → 11 → 10 → 1 → 9 → 3 → 12 → 4 → 15 → 14 → 0`
  - travel = **31**，penalty = **84090**，**J = 84121**
  - **与纯 Python v2 路径完全一致，ΔJ = 0**——构成跨独立工具的最强一致性证据
  - SDK baseline 0.07s 找到 1/50 可行；tuned 3.85s 找到 6/1000 可行（feas_rate ≈ 0.6%）；与 Q1 SDK 同结构性瓶颈
- **τ̂_p (位置 p 的预期到达时刻)** = [2.8, 8.97, 15.14, 21.31, 27.49, 33.66, 39.83, 46.0, 52.17, 58.34, 64.51, 70.69, 76.86, 83.03, 89.2] ——位置 9 起 τ̂_p > 30 已超过大多数客户的 b_i，结构性晚到不可避免

### 消融实验：方案 D（已记录失败原因）

- **配置**：方案 C 同套 SA 参数（5 seeds × 200 chains, T0=500, α=0.995, iter/T=80）
- **失败结果**：
  - **可行率 0/1000 = 0.00%**（5 个 seed 全军覆没）
  - 方案 D **对角线最大时间窗罚 = 119,197**，A_pen = 200，**比值 = 596×**
  - SA 优先消除巨大对角项（即让 x[i,p]=0），导致约束 `∑x[i,p]=1` 全部失败 → one-hot 不可行
- **诊断与论文价值**：方案 D 设计上看似"在 QUBO 内含时间窗信息"，实操上因 τ̂_p (后位置) ≈ 89 远超时间窗 b_i ≤ 30，把对角线项推到 10⁵ 量级，淹没 one-hot 罚 → 这正是铁律 §二.5 警示的"**非必要不引入辅助比特/罚项**"。**该消融恰好论证了方案 C 的合理性**：用更少的"罚项耦合"得到了更高的可行率与最优 J。
